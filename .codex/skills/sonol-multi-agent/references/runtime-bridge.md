# Runtime Bridge

`sonol-multi-agent` currently defaults to a Codex adapter unless the
environment explicitly overrides it, but the runtime bridge should be
treated as adapter-backed Sonol orchestration.
Sub-agents do not invent their own status protocol.

This bridge assumes only a public sub-agent control surface. It relies
on prompt packets, shared artifacts, and runtime events. It does not assume a
public low-level API exists for direct thread handles or terminal injection.
Its preferred reporting path comes from the generated packet. `script_commands`
is the common default, and `json_ingest` is the adapter-neutral fallback or
provider-specific preferred mode when the packet says so.

If Sonol uses a hosted plan normalizer, keep that split narrow: the local
Codex/Claude session authors the creative draft, the hosted service only
normalizes and binds it, and runtime event ownership remains in the local
SQLite-backed Sonol workflow unless an explicit transport upgrade is
introduced.

## Required sub-agent instructions

Each sub-agent should be told:

1. Load the current Sonol runtime reporting helper or use the generated JSON ingest path.
2. Respect the provided control profile:
   - `purpose`
   - `provider_agent_type`
   - current adapter-specific alias fields if present
   - `custom_agent_name`
   - `custom_config_file`
   - `developer_instructions`
   - `model`
   - `model_reasoning_effort`
   - `sandbox_mode`
   - `mcp_servers`
   - `skills_config`
   - `approval_mode`
   - `communication_mode`
   - `read_paths`
   - `write_paths`
   - `deny_paths`
   - `operational_constraints`
   - `depends_on`
   - `reporting_contract`
3. Do not spawn child sub-agents in v1.
4. Use Main Agent mediation rather than hidden peer-to-peer messaging.
5. Expect follow-up only through the main conversation or `/agent`, not through arbitrary external thread injection.
6. Use the transport named in the generated packet.
   If the packet says `preferred_mode=json_ingest`, use the generated JSON
   ingest path first. If it says `preferred_mode=script_commands`, use direct
   `report-*` helper-script writes first.
7. Read generated runtime context files instead of repeating long setup instructions.
8. Report `progress_event` at task start and after each meaningful milestone.
9. Report `artifact_event` when creating outputs worth inspection.
10. Report `completion_event` at task completion.
11. Report `session_updated` when blocked or idle. Use `completion_event` as
    the normal terminal success or failure report.
12. Keep event text short and factual.
13. Receive a concrete prompt packet generated by `build-subagent-prompt.mjs`.
14. Never receive an ad-hoc freeform role prompt when a run already exists.
15. The prompt packet must be generated from the current run and exact planned `agent_id`.
16. Prefer the generated run-scoped prompt file under `<runtime_root>/<run_id>/prompts/<agent_id>.txt`
    so the delegated text matches the active run exactly.
17. Treat `<runtime_root>/<run_id>/prompts/<agent_id>.txt` as the authoritative runtime prompt path.
    Do not fall back to any mutable plan-level alias once a concrete run prompt exists.
17.5. In a manifest-only adapter, do not replace that file with a new free-form
    provider prompt. Copy or pass the run-scoped prompt text exactly.
18. Every delegated sub-agent must reuse the exact `plan_id`, `run_id`, and
    `agent_id` from that packet in all `report-*` calls.
19. If the packet includes `assistant_launch_recipe`, prefer it as the exact
    launch recipe for the fields the host surface can actually enforce. Other
    Sonol controls such as path policy, MCP surface, and helper-skill policy
    remain coordination rules unless the host surface enforces them natively.
20. In manifest-only adapters, `fork_context=false` is a launch invariant.
    Do not rebuild the launch in a way that inherits stale parent context.

The prompt packet should make the following explicit:

- `provider_agent_type`
- current adapter-specific base type aliases if present
- `purpose`
- expected deliverables
- dependency and wait policy
- model and reasoning effort
- sandbox mode
- allowed MCP and skills surface
- Sonol path and operational policy

## Required identifiers

Every sub-agent prompt should include:

- `run_id`
- `agent_id`
- current `task_id`
- `purpose`
- `depends_on`
- `approval_mode`
- `communication_mode`
- control limits for paths and reporting
- `schema_version` currently `1.0.0`
- explicit `report-progress`, `report-session`, and `report-completion` commands
  with the exact current `plan_id`, `run_id`, `agent_id`, and `task_id` where
  that event family requires one

## Minimal reporting pattern

1. Emit progress at task start
2. Emit progress after each major step
3. Emit artifact when a file or summary is ready
4. Emit `session_updated` when blocked or when waiting idle for the next unit
5. Emit `completion_event` at the end

If a sub-agent reports under a different run than the active run, Sonol should
consider that a coordination failure, not a valid report for the dashboard.
The runtime report path should reject that stale run id and point the operator
to the currently advertised recovery prompt path. In the current implementation,
stale-run hints resolve to the run-scoped prompt file under
`<runtime_root>/<run_id>/prompts/<agent_id>.txt`.
Diagnostics should flag this as `reports_written_to_other_run`.

## Unsupported assumptions

- direct dashboard launch of a multi-agent run
- direct text injection into an existing terminal session
- undocumented `spawn`, `wait`, `list`, `terminate`, or `send_input` controls
- nested autonomous child-agent trees in v1

## Replacement patterns

- steer a sub-agent through the Main Agent or `/agent`
- make controls stricter with `developer_instructions`
- reduce `sandbox_mode`, `mcp_servers`, or `skills_config`
- use runtime events for observability instead of direct sub-agent control
- use `show-run-launch-manifest.mjs` or `/api/runs/<run_id>/launch-manifest`
  to obtain exact launch-ready packets when a human or Main Agent must launch
  real provider sub-agents
- when available, use `assistant_launch_recipe.tool_args` exactly as emitted
  by the packet instead of rebuilding a tool call manually
- treat sandbox mode, MCP surface, helper-skill requirements, and path policy
  as Sonol coordination constraints unless the host launch surface enforces them
  natively
